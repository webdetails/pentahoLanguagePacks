# Maven Conversion: pentahoLanguagePacks

## Summary

The pentahoLanguagePacks project has been converted from an Ant/Ivy build to a Maven build, following the same patterns used by CPK and Sparkl.

## What Was Removed

The following files have been **deleted** from the project:

| File | Original Purpose | Why It's Gone |
|------|-----------------|---------------|
| `build.xml` | Ant build: `dist`, `build-language-pack`, `install-plugin` | Replaced by Maven assembly + antrun profiles |
| `build.properties` | Ivy/Ant config (version, artifact IDs, CPK/CPF revisions) | Replaced by `pom.xml` properties |
| `ivy.xml` | Declared runtime JARs (cpf-core, cpk-core, jackson, xmltask) | Replaced by Maven `<dependencies>` |
| `ivysettings.xml` | Pointed to Pentaho Nexus for Ivy resolution | Maven uses standard repository config from parent POM |
| `build-res/` | Pentaho subfloor build scripts (subfloor.xml, subfloor-pkg.xml) | Not needed with Maven |
| `lib/` (resolved JARs) | JARs downloaded by `ant resolve` | Maven assembly places JARs in `lib/` from dependency resolution |
| `.ant-targets-build.xml` | Ant IDE target cache | Not needed |
| `.travis.yml` | Travis CI config | Replaced by modern CI (GitHub Actions, Jenkins, etc.) |
| `pentahoLanguagePacks.iml` | IntelliJ module file | Generated by IDE, gitignored |
| `version.xml` (root) | Hardcoded version file | Generated from template via Maven resource filtering |
| `cpk.spring.xml` (root) | Spring bean config | Moved to `src/main/resources/` |
| `plugin.spring.xml` (root) | Spring plugin config | Moved to `src/main/resources/` |

### Why Ivy Was Never Needed at Runtime

Sparkl's `updateLibs.kjb` ‚Äî the mechanism that provisions JARs to plugins on a running server ‚Äî performs a simple **file copy** from `${cpk.lib.dir}` (Sparkl's own `lib/` directory) to `${pluginDir}/lib`. It never invokes Ant, Ivy, or any build tool. The Ivy/Ant files were purely a developer convenience for offline JAR resolution, which Maven now handles better.

## How to Build

### Build the full plugin zip

```bash
mvn clean package
```

**Output:** `assemblies/pentaho-plugin/target/language-pack-installer-pentaho-plugin-11.1.0.0-SNAPSHOT.zip`

This zip contains:
- All plugin resources (cpk.xml, plugin.xml, dashboards, endpoints, data, static, resources)
- Runtime JARs in `lib/` (cpf-core, cpf-pentaho, cpk-core, cpk-pentaho5)
- Ready to deploy to `pentaho-solutions/system/languagePackInstaller/`

### Build a standalone language pack for a specific locale

```bash
mvn clean package -DlangCode=pt_PT
```

**Output:**
- `assemblies/pentaho-plugin/target/dist/languagePack_pt_PT-<version>.zip` ‚Äî standalone installer

Note: When `-DlangCode` is set, the base assembly plugin is skipped (phase set to `none`) to avoid redundant work.

### Build standalone installers for ALL locales

```bash
mvn clean package -Pbuild-all-language-packs
```

**Output:** One zip per locale in `assemblies/pentaho-plugin/target/dist/`

This uses the `iterator-maven-plugin` which scans each subdirectory in `data/` and forks a Maven `package` invocation with `-DlangCode=<locale>` for each one. The `en` locale is excluded.

## Key Design Decisions

### Templates in `src/main/resources-filtered/`

The Pentaho parent POM (`pentaho-ce-jar-parent-pom`) already configures `src/main/resources-filtered/` as a filtered resource directory. Templates placed here are automatically processed by `maven-resources-plugin` during the build.

All template tokens use standard Maven `${property}` syntax. Pentaho **runtime** resource-bundle expressions (e.g., `${Launcher.menuLanguagePacks}`, `${languagePack.title}`) are escaped as `\${...}` in the templates. The `maven-resources-plugin` is configured with `<escapeString>\</escapeString>` so these are emitted as literal `${...}` in the output for Pentaho to resolve at runtime.

`src/main/resources/` contains only files that have **no per-locale variant**: `cpk.spring.xml`, `dashboards/admin/main.wcdf`, `endpoints/`, `static/`, and `resources/`. Everything else lives exclusively in `resources-filtered/` and is parameterised by `${langpack.name}` (and related properties). For the base installer build, the `main-build-defaults` profile supplies `langpack.name=languagePackInstaller`, `langpack.title=Language Pack Installer`, etc., so no separate static copies are needed.

### Why Ant Is Still Used for Per-Locale Builds

The per-locale build involves operations that have no clean declarative Maven equivalent:

1. **JSON metadata parsing** ‚Äî Reads `metadata.json` via Groovy `JsonSlurper` (embedded in Ant `<script>`) to extract structured fields (maintainer, maturity, version, etc.)
2. **Token replacement in non-resource files** ‚Äî Patches dashboard `.cdfde` files and `plugin.spring.xml` to rename plugin references
3. **Conditional file inclusion** ‚Äî Only includes screenshots/flags if they exist
4. **Text replacement in `cpk.xml`** ‚Äî Updates name/description from locale messages

The Ant logic lives in a separate file (`src/build/build-language-pack.xml`) keeping the POM clean. The POM profile is just a thin wrapper that passes Maven properties and invokes the Ant file.

### xmltask Removed

The original build used `com.oopsconsultancy:xmltask` for XPath-based XML editing of `cpk.xml`. This artifact is not available in Maven Central or the Pentaho repository. Since the edit only changes two known text nodes (`<name>` and `<description>` inside `<cpk><info>`), it was replaced with simple Ant `<replace>` token matching against the known default values.

### Iterator Maven Plugin for Build-All

Instead of a shell script or an Ant `<exec>` loop, the `build-all-language-packs` profile uses `com.soebes.maven.plugins:iterator-maven-plugin:0.5.1`. This keeps everything in Maven-land:

```xml
<folder>${data.dir}</folder>
<excludes><exclude>en</exclude></excludes>
<itemVariableName>langCode</itemVariableName>
```

Each iteration forks a Maven invocation of `package` on the same module. Results accumulate safely because each locale writes to its own output path (`target/langpack/languagePack_<locale>/` and `target/dist/languagePack_<locale>-*.zip`).

### Groovy for JSON Parsing

The original Ant build used inline JavaScript with `eval()` to parse `metadata.json`. Since Java 15+ removed Nashorn, we use Groovy's `JsonSlurper` instead ‚Äî safer, cleaner, and a first-class Maven citizen. Groovy is provided as a plugin dependency of `maven-antrun-plugin` (`org.apache.groovy:groovy-all:4.0.24`), requiring no separate plugin or build phase.

## Dependency Management

Dependencies are declared in the root `pom.xml` under `<dependencyManagement>`:

```xml
<dependency>
  <groupId>pentaho</groupId>
  <artifactId>cpf-core</artifactId>
  <version>${pentaho-cpf-plugin.version}</version>
</dependency>
<dependency>
  <groupId>pentaho</groupId>
  <artifactId>cpf-pentaho</artifactId>
  <version>${pentaho-cpf-plugin.version}</version>
</dependency>
<dependency>
  <groupId>org.pentaho.ctools</groupId>
  <artifactId>cpk-core</artifactId>
  <version>${pentaho-cpk-plugin.version}</version>
</dependency>
<dependency>
  <groupId>org.pentaho.ctools</groupId>
  <artifactId>cpk-pentaho5</artifactId>
  <version>${pentaho-cpk-plugin.version}</version>
</dependency>
```

The plugin assembly module uses these (without versions) and the assembly descriptor places them in `lib/`.

For per-locale builds, `maven-dependency-plugin:copy-dependencies` runs in `prepare-package` phase to copy the CPK/CPF jars to `target/lib/`, which the Ant build then copies into the staged plugin's `lib/` folder.

## Migration from Old Workflow

| Old (Ant/Ivy) | New (Maven) |
|---------------|-------------|
| `ant resolve` | `mvn package` (implicit) |
| `ant dist` | `mvn clean package` |
| `ant build-language-pack -DlangCode=pt_PT` | `mvn clean package -DlangCode=pt_PT` |
| `ant build-language-packs` | `mvn clean package -Pbuild-all-language-packs` |
| `ant install-plugin` | Deploy zip to `pentaho-solutions/system/` manually or via CI |
| Edit `build.properties` for versions | Edit `pom.xml` `<properties>` section |
| `ivy-nexus.pentaho.org` | Standard Maven repos from parent POM |
| `lib/xmltask-1.16.jar` (checked in) | Removed ‚Äî replaced with Ant `<replace>` |
| Nashorn (bundled in JDK < 15) | `org.apache.groovy:groovy-all:4.0.24` as antrun plugin dependency |

## Validation Tests

The module includes JUnit 5 parameterized tests (`src/test/java/org/pentaho/ctools/langpack/LanguagePackValidationTest.java`) that validate all locale data:

- `metadata.json` exists and is valid JSON for every locale
- `messages.properties` exists and contains required keys (`languagePack.title`, `languagePack.description`)
- Each locale (except `en`) has at least a `system/` or `tomcat/` directory
- Minimum locale count (40+)

Run with: `mvn test -pl assemblies/pentaho-plugin`

## Continuous Integration

Three GitHub Actions workflows cover the full lifecycle:

| Workflow | Trigger | Maven version | Deploys to Artifactory? | GitHub Release? |
|---|---|---|---|---|
| `pr.yml` | pull_request ‚Üí master | SNAPSHOT (unchanged) | ‚ùå | ‚ùå |
| `merge.yml` | push ‚Üí master | SNAPSHOT | ‚úÖ SNAPSHOT repo (transient) | ‚ùå |
| `release.yml` | `workflow_dispatch` (manual) | Set to input (e.g. `11.1.0`) | ‚úÖ Release repo (permanent) | ‚úÖ with all ZIPs |

### `pr.yml` ‚Äî pull requests
Runs `validate` (JUnit tests across all 43+ locales) + `build-installer` + `build-one-locale` (builds `pt_PT` end-to-end as a representative smoke test). No deployment.

### `merge.yml` ‚Äî push to master
1. `validate` ‚Äî run JUnit tests
2. `setup-locale-matrix` ‚Äî scans `data/` at runtime, outputs a JSON array of locale codes (no hardcoded list)
3. `deploy-installer` ‚Äî `mvn deploy` ‚Üí publishes main installer ZIP as SNAPSHOT to Artifactory
4. `deploy-locale` matrix ‚Äî one parallel job per locale: `mvn deploy -pl assemblies/pentaho-plugin -DlangCode=<locale> -DskipTests` ‚Äî each locale built exactly once and published as SNAPSHOT

### `release.yml` ‚Äî manual release (workflow_dispatch)
Takes a `version` input (e.g. `11.1.0`). In each job, runs `mvn versions:set -DnewVersion=<version>` before deploying, which removes `-SNAPSHOT` so Artifactory stores the artifacts permanently in the release repository.

1. `setup-locale-matrix` ‚Äî same dynamic locale discovery
2. `deploy-installer` ‚Äî set version + `mvn deploy` ‚Üí permanent release artifact in Artifactory
3. `deploy-locale` matrix ‚Äî set version + `mvn deploy -DlangCode=<locale>` for each locale in parallel
4. `github-release` ‚Äî waits for all jobs, downloads all ZIPs, creates a GitHub Release tagged `v<version>` with all locale ZIPs and the installer attached

A `dry_run` boolean input (default: `true`) runs the full build but skips Artifactory deploy and GitHub Release creation, for testing the release pipeline safely.

All workflows use `pentaho/actions-common@stable` to retrieve the Maven `settings.xml` (providing repository access and deploy credentials), `actions/checkout@v6`, `actions/setup-java@v4`, and `actions/upload-artifact@v7`.

### How locale ZIPs become deployable Maven artifacts

The `build-language-pack` profile (activated by `-DlangCode`) assembles the locale ZIP and then uses `build-helper-maven-plugin:attach-artifact` to attach it with the locale code as classifier:

```
org.pentaho.ctools:language-pack-installer-pentaho-plugin:<version>:zip:pt_PT
org.pentaho.ctools:language-pack-installer-pentaho-plugin:<version>:zip:fr
...
```

This means a standard `mvn deploy -DlangCode=XX` is all that's needed ‚Äî no special flags. The CI workflows use this directly in the matrix jobs.

### Local build reference

```bash
mvn package -DlangCode=pt_PT                 # build one locale (no deploy)
mvn deploy  -DlangCode=pt_PT                 # build + deploy one locale
mvn package -Pbuild-all-language-packs       # build all locale ZIPs locally (serial)
mvn test    -pl assemblies/pentaho-plugin    # validate all locale data
```
